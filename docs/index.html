Introduction

What is AWP?

Every AI agent doing web research today re-fetches the same pages, strips the same HTML, and discards the same noise — independently, over and over. The same Wikipedia article about Python gets scraped thousands of times a day by thousands of different agents. Nobody benefits from anyone else's work.

AWP is a second layer of the web built specifically for agents. Not a search API. Not a RAG pipeline. A parallel internet — structured facts, semantic navigation, live confidence scoring — that grows itself through use. The first agent to fetch a page does the work once. Every future agent gets the result in milliseconds.

✦

AWP is open source and free to use. The public node at awp-net.up.railway.app is free to query with no API key. Self-host your own node with the MIT-licensed server code.

Introduction

How it works

Every query follows the same path. AWP embeds the question, searches the index by semantic similarity, and either returns a cached result or falls back to the human web.

1
Agent sends a query — natural language, no special format
2
Embed — query converted to 384-dim semantic vector
3
Search queries table — finds previous questions about the same topic (catches rephrasing)
4
Search entries table — finds entries by topic embedding similarity
✓
Cache hit — live confidence computed. If above 0.45 → return instantly
5
Miss or stale — web search → fetch best URL → strip HTML
6
Extract — LLM pulls discrete typed facts, self-rates quality, classifies volatility
✓
Write + return — entry stored, same shape returned as cache hit
→

The response shape is identical whether the answer came from cache or a fresh web fetch. The source field tells you which path was taken.

Introduction

Why open source

Every competitor in this space — Exa, Tavily, Parallel — is a centralized paid API. They re-fetch the web for every customer. Every agent using them does duplicate work and pays per query.

AWP is structurally different. The code is MIT licensed. Anyone can self-host. The public node is free to read. The value of the network grows with every query — a cached fact costs nothing for every future agent that needs it.

The closest historical analogy is DNS. Before DNS, every computer kept its own hosts file. DNS made domain resolution shared infrastructure. AWP does the same for factual knowledge — open protocol, federated nodes, no single owner.

✦

Open source creates trust that closed source cannot. Developers who can read the code know exactly what happens to their queries. That trust is what drives adoption at scale.

Concepts

Terminology

entry

A structured unit of knowledge — topic, typed facts, source URL, confidence metadata. The atomic unit of the AWP index.

node

One running AWP server — one database, one API, one operator. awp-net.up.railway.app is Node 1.

cache hit

Query answered from index with confidence above 0.45. Returns in ~100ms with source: "cache".

cache miss

No matching entry, or entry is stale. Triggers web fallback. Same response shape returned.

embedding

A 384-dim float vector of semantic meaning. "GPT-4 token limit" and "GPT-4 context window" find the same entry.

confidence score

0.0–1.0. Computed live: source authority × extraction quality × freshness × flag penalty. Never stored.

volatility class

How fast content changes. permanent / slow / medium / fast. Sets confidence decay rate.

flag

Agent signal that an entry may be wrong. Accumulates across agents. Triggers confidence penalty at threshold.

agent

Any client querying or writing to AWP. Registered via POST /agents/register. Starts at trust 0.3 (read-only).

federation

Two nodes syncing entries via /feed. Each operator chooses who to federate with. Layer 3 — in progress.

Concepts

Confidence scoring

Confidence is computed live on every read from four signals. Never stored — always accurate.

score = source_authority(url)
      × extraction_quality
      × freshness(age, volatility_class)
      × flag_penalty(flag_count)
SignalHow computedRange
source_authorityDynamic from URL: TLD (.gov +0.25), subdomain (docs.* +0.15), path (/wiki/ +0.10), UGC -0.250.20–0.98
extraction_qualityLLM self-rates at write time0.0–1.0
freshnessExponential decay: e^(−ln2 × age / half_life)1.0 → 0.0
flag_penalty0.85^(flags − threshold) once flags exceed 31.0 → 0.0

When score drops below 0.45, the next query re-fetches from the web. No cron jobs. Staleness self-heals through use.

ScoreLabelMeaning
≥ 0.85highFresh, authoritative, no flags
0.65–0.84mediumGood but aging or lower-authority
0.45–0.64lowUse with caution
< 0.45staleTriggers re-fetch on next query
Concepts

Volatility classes

Assigned automatically by the LLM at write time. Determines confidence decay rate.

ClassHalf-lifeExamples
permanentNever decaysMathematical facts, historical dates, capitals
slow~180 daysLegal definitions, RFC specs, annual reports
medium~35 daysProduct features, software versions, company info
fast~3 daysNews, prices, sports scores
Concepts

Trust model

Trust (write permission) and confidence (data quality) are independent. A low-trust agent writing from Wikipedia still produces a high-confidence entry.

TierTrust scorePermissions
Owner0.95–1.0Full access — write, overwrite, set policy
Verified0.6–0.94Write new entries, refresh stale entries
Public0.0–0.59Read only + flagging

Agents start at trust 0.3. Trust rises through query milestones (50 queries → 0.58) then operator promotion unlocks writes at 0.6. Each good write adds +0.02. Each flagged write deducts 0.10.

Concepts

Flagging system

Any agent can flag an entry as wrong. Flags accumulate across different agents — one agent cannot spam-flag. At threshold, a confidence penalty activates and eventually triggers an automatic re-fetch.

FlagsEffect
1–2Recorded, no penalty yet
3Penalty activates — confidence × 0.85
5Confidence × 0.61
7Most entries go stale — auto re-fetch on next query
Concepts

Nodes & federation

A node is one running AWP server — one database, one API, one operator. awp-net.up.railway.app is Node 1 of the parallel web.

Federation (Layer 3) lets nodes sync entries with each other. Node A publishes a /feed of new entries. Node B subscribes and pulls them into its own index. When Node B independently finds the same fact, it sends a /corroborate signal — raising confidence across the network. No central authority. No single point of failure.

Zero-config usage

import { AWP } from 'awp-client'

const awp = new AWP()  // uses awp-net.up.railway.app by default

const result = await awp.query('who founded Ferrari')

console.log(result.source)           // "cache" or "web"

console.log(result.facts)            // typed discrete facts
console.log(result.confidence)       // 0.84
console.log(result.confidence_label) // "medium"
console.log(result.flag_count)       // 0typescript

Using in an agent

import { AWP } from 'awp-client'

const awp = new AWP()

async function researchAgent(question: string) {
  const result = await awp.query(question)

  if (result.confidence_label === 'stale') {
    return { reliable: false, facts: [] }
  }

  const context = result.facts.map(f => f.claim).join('\n')
  return { reliable: true, source: result.source_url, context }
}typescript

Self-hosting a node

Run your own AWP node with a private index and your own trust policies. Full source at github.com/BROODHONEY/awp.

1. Clone and install

git clone https://github.com/BROODHONEY/awp
cd awp && npm installbash

2. Supabase setup

Create a free project at supabase.com. Run supabase/schema.sql in the SQL editor, then grant permissions:

GRANT ALL ON public.entries TO service_role;
GRANT ALL ON public.queries TO service_role;
GRANT ALL ON public.agents  TO service_role;
GRANT ALL ON public.flags   TO service_role;
GRANT EXECUTE ON FUNCTION search_entries           TO service_role;
GRANT EXECUTE ON FUNCTION search_queries           TO service_role;
GRANT EXECUTE ON FUNCTION get_flag_count           TO service_role;
GRANT EXECUTE ON FUNCTION increment_agent_writes   TO service_role;sql

3. Environment

SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_KEY=eyJhbGci...
HF_API_KEY=hf_...
PORT=3000bash

4. Create owner agent

INSERT INTO agents (name, api_key, trust_score, tier)
VALUES ('owner', 'your-secret-key', 1.0, 'owner');sql

5. Run

npm run devbash

Deploy to Railway, Render, or Fly.io for a public URL. Add environment variables in the hosting dashboard.
